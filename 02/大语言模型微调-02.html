CHAPTER 01

全参数微调的算力瓶颈

在上节课中，我们从物理与数学层面剖析了大型语言模型的本质。大模型是一个由海量参数构成的复杂数学函数。在"预训练"阶段，模型通过大规模计算海量文本，确立了基础的参数数值，从而具备了语言能力与常识。

而"微调"则是利用结构化的特定领域数据，计算输出结果与标准答案的误差，并通过反向传播永久性地修改模型内部一小部分权重的数值。

既然微调需要计算梯度并更新底层参数，这必然带来一个严峻的问题：修改模型所有参数的"全参数微调"需要极高的显存和算力，个人或普通企业难以承受。

显存需求对比
全参数微调（7B 模型） 60 ~ 80 GB
LoRA 微调（消费级显卡） 16 GB

* 以 7B 参数模型、FP16 精度、批大小 1 为基准估算

全参数微调

全参数微调，顾名思义，要求模型内部所有的权重参数都参与反向传播的计算与数值更新。在显卡的显存中，这一过程会产生极其庞大的数据堆积。

1. 静态参数与激活值的显存占用

首先，模型自身的数十亿个基础参数需要被完整地加载到显存中。以一个标准的 70 亿参数（7B）模型为例，如果使用 16 位浮点数精度（FP16）存储，仅这些静态参数本身就会占据约 14GB 的物理显存。

其次，在数据进行前向传播计算时，为了能够在后续准确执行反向传播，显卡必须将每一层计算产生的巨大中间结果（称为"激活值"）全部临时存储在显存中。这部分的显存占用量会随着输入文本的长度呈指数级增长。

2. 梯度矩阵的显存翻倍

在全参数微调中，系统需要为模型中的每一个参与更新的参数计算出一个对应的梯度。如果模型有 70 亿个参数，显卡就必须在显存中开辟出与原参数矩阵完全一样大的新空间，专门用于存放这些梯度数值。这直接导致显存需求在原有基础上再次翻倍。

3. 优化器状态的巨大消耗

现代深度学习通常采用复杂的优化器（如 Adam），它们不仅需要当前的梯度，还需要记录每一个参数在过去多次计算中的历史梯度信息。对于每一个参数，优化器通常需要额外存储两个甚至更多的高精度浮点数状态变量。

综合来看，一个原本只需 14GB 显存即可勉强运行对话的 7B 模型，如果进行全参数微调，其显存峰值需求往往会飙升至 60GB 甚至 80GB 以上。这就要求必须使用多张极其昂贵的顶级企业级显卡（如 NVIDIA A100 或 H100）才能完成计算。

本征维度

面对全参数微调对物理资源的极端挑战，研究视角重新回归数学底层逻辑，并提出了一个核心工程假设：

大型语言模型在执行特定任务的微调时，其参数更新其实存在一个极低的"本征维度"。

"本征维度"可以理解为：真正有效、起作用的变化方向有多少个。

你看到的大模型可能有几十亿个参数，但在实际做某个具体任务时，并不需要在所有这些参数上都做复杂调整。真正影响模型性能提升的，只是其中一小部分"关键方向"。

类比理解：想象你在调一个复杂的音响系统，面板上有 1000个旋钮（参数），但实际上，只需要调 5~10个关键旋钮，就能让音质明显变好。这 5~10 个关键旋钮的数量，就类似"本征维度"。

这就引出了 LoRA 的核心架构设计：既然绝大部分的参数修改是冗余的，我们完全可以绕开那个庞大的原始矩阵，用一种维度极低的数学结构来精确模拟这种微小的参数变化。

LoRA 的核心物理架构

基于本征维度的发现，LoRA 技术在显存中构建了一套全新的并行计算网络。它的物理架构主要由两个截然不同的操作步骤组成。

基础参数的绝对冻结

当基座模型加载到显存后，系统首先执行的第一条指令就是"冻结"原始模型内部所有的基础参数。

在物理计算层面上，"冻结"意味着系统明确切断了误差向这些原始参数传递的计算路径。在随后的所有反向传播过程中，显卡不再为这数百亿个基础参数计算梯度，也不再存储它们的历史梯度和优化器状态。

这一简单的物理阻断操作，直接将微调过程中最庞大的显存开销（梯度矩阵和优化器状态矩阵）瞬间清零。被冻结的基础参数在微调过程中，仅仅作为固定的数据通道参与前向传播，其自身的浮点数值从始至终绝对不会发生任何改变。

附加参数模块的注入

在彻底冻结了庞大的主体结构后，系统会在原有的网络层旁边，额外并联添加一个极小规模的"附加参数模块"。

这个附加模块是全新生成的。它的核心特征在于采用了两个具有极端不对称形状的小型矩阵进行乘法组合。在随后的微调训练阶段，显卡的全部算力、所有的误差反向传播计算以及最终的数值修改，将 100% 集中在这两个额外添加的小型矩阵上。

降维与升维：低秩矩阵的数学拆解

LoRA 运用了线性代数中的矩阵分解技术，即通过降维和升维的两次连续计算，来替代原始的庞大计算。

相比于全参数微调需要计算和存储的 1677 万个参数，LoRA 将实际需要计算的参数量急剧缩减了 99.6% 以上。这让普通企业甚至个人开发者，依靠单张家用级的中高端显卡（如 RTX 3090 或 4090），就能顺利完成百亿参数规模大模型的微调计算。

训练阶段的双路数据流转

1. 前向传播：分流与汇总

当代表输入文本的高维向量到达模型的某一个计算层时，数据流会在物理层面被一分为二：

  • 主干路计算：输入向量穿过被完全冻结的原始矩阵 W，进行一次巨大的矩阵乘法，得出一个包含 4096 个数值的基线结果向量。
  • 旁路计算（LoRA 模块）：同一个输入向量同步进入 LoRA 的附加模块。首先与降维矩阵 A 相乘，被压缩成 8 个数值；紧接着与升维矩阵 B 相乘，重新膨胀回包含 4096 个数值的调整向量。
  • 向量叠加：系统将主干路产生的"基线结果向量"与旁路产生的"调整向量"进行简单的对应位置数值相加，作为最终计算结果继续向模型的下一层传递。

2. 反向传播：精准定位修改

由于主干路的原始矩阵 W 被设置了"冻结"指令，误差信号在传递到这里时会被直接阻断，不会产生任何梯度计算。

所有的误差信号和梯度计算算力，将全部精准地导入到那两条旁路分支中。优化器算法仅仅根据误差，极其迅速地计算出如何微调矩阵 A 和矩阵 B 中那区区 6.5 万个浮点数的数值。

部署阶段的物理合并机制

当所有数据的训练循环宣告结束，在我们的硬盘上，除了原本的基座大模型文件外，还会额外生成一个体积非常小的文件，这个文件专门存储了训练好的所有矩阵 A 和矩阵 B 的最终数值，在工程上被称为"LoRA 权重文件"。

为了避免推理延迟，在正式部署前，我们需要执行一个关键的物理操作：模型权重合并。

A (4096×8) × B (8×4096) → ΔW (4096×4096)
W_new = W_frozen + ΔW

合并带来的架构复原

完成加法覆写后，物理层面上的 LoRA 旁路模块（矩阵 A 和矩阵 B）被彻底销毁和移除。合并后的新模型在物理结构和层级数量上，与微调前最初始的基座模型完全一模一样，没有任何额外的旁路分支。

这种合并机制带来了极为重要的工程优势：微调后的模型在实际运行时，其计算复杂度和文本生成速度与原始基座模型完全一致，不存在任何因为微调而增加的性能损耗。

CHAPTER 03

搭建本地微调运行环境

在掌握了 LoRA 这种通过引入低维度旁路矩阵来降低算力消耗的高效微调架构之后，我们已经可以在消费级显卡上搭建起微调的运行环境，环境包括硬件环境和软件环境。

CUDA 运行时的验证与环境映射

硬件就不多说了，显存越大越好，显卡越高端越好，建议至少 16G 显存。

CUDA 是由 NVIDIA（英伟达）推出的一种并行计算平台和编程模型。在如今的大模型微调和训练中，CUDA 几乎是不可或缺的底层基础设施。简单来说，它是一座桥梁，让我们能够直接利用 NVIDIA GPU 的强大算力，去解决复杂的通用计算问题。

打开 Windows 终端，输入以下命令查看 CUDA 版本：

nvidia-smi

在输出的表格右上角会显示 CUDA Version: 13.x。需要注意，此处显示的是显卡驱动程序支持的最高 CUDA 版本，最低不能低于 12.6 版本，因为我们使用的深度学习框架 PyTorch 支持的最低 CUDA 版本是 12.6。

基于 UV 的 Python 隔离环境构建

Python 是深度学习的最佳语言，因此为了避免不同项目之间产生的依赖库冲突，必须为微调任务构建完全独立的虚拟环境。

uv 是一个由 Rust 语言编写的极速 Python 包和环境管理器。与传统的 pip 或 conda 相比，uv 能够并发解析依赖树，其环境创建和包安装速度有数量级的提升。

1
新建工作目录

在硬盘的根目录或工作目录下，新建一个专门用于微调的文件夹（例如 D:\LLM_Workspace）。

2
创建并激活虚拟环境
# 创建虚拟环境 uv venv # 激活环境（Windows） .venv\Scripts\activate

当命令行提示符前出现 (LLM_Workspace) 字样时，表明系统已成功切入该隔离环境。

深度学习框架 PyTorch 安装

PyTorch 是整个微调系统的核心计算框架，它向下负责调用 CUDA 与显卡进行通信，向上为微调工具提供 Python 操作接口。

前往 pytorch.org，选择适合您 CUDA 版本的安装命令。以下以 CUDA 13.0 为例：

uv pip install torch torchvision --index-url https://download.pytorch.org/whl/cu130

uv 会自动解析环境、下载并安装核心包。由于 uv 采用了全局缓存机制，如果后续在其他文件夹重新建立虚拟环境并安装相同的包，无需重新下载。

测试环境

完成上述所有配置后，需要通过一段 Python 代码进行一次测试，以验证硬件、驱动、加速库和框架之间的数据通路是否已完全打通。

import torch # 1. 验证 PyTorch 是否成功探测到底层 CUDA 驱动 print("CUDA Available:", torch.cuda.is_available()) # 2. 验证是否准确识别到当前的物理显卡型号 print("Device Name:", torch.cuda.get_device_name(0)) # 3. 验证显卡的硬件计算能力架构 print("Compute Capability:", torch.cuda.get_device_capability(0)) # 4. 验证 BF16 (BFloat16) 数据类型的硬件原生支持 # 这是进行高效大模型微调的关键特性 print("BF16 Support:", torch.cuda.is_bf16_supported())
python .\test.py

如果所有输出均返回预期结果（即 CUDA Available: True，Device Name 准确显示显卡型号，且 BF16 Support: True），则证明系统已具备运行微调的标准条件。

CHAPTER 04

部署 LLaMA-Factory

什么是 LLaMA-Factory

大语言模型的微调是一个极其复杂的工程过程，涉及海量矩阵的梯度计算、优化器状态的显存调度、并行计算的数据切分等。LLaMA-Factory（大模型微调工厂）是一个高度集成的全栈式微调工程框架，它改变了以往需要手动编写成百上千行 Python 脚本的现状。

底层算子与高级库的封装整合

LLaMA-Factory 在底层严格依赖 PyTorch 框架，并在其上无缝集成了 Hugging Face 生态系统中最核心的四个开源组件：

transformers peft trl accelerate
  • transformers：负责加载各类开源模型（Llama、Qwen、ChatGLM 等）的标准网络结构与权重文件。
  • peft：提供 LoRA、QLoRA 等参数高效微调算法的标准数学实现。
  • trl：提供针对大模型的强化学习对齐算法（如 PPO、DPO）实现。
  • accelerate：负责处理张量在 CPU 内存与 GPU 显存之间的物理调度与设备映射。

可视化交互界面（Web UI）

通过集成 Gradio 库，LLaMA-Factory 提供了一个运行在浏览器中的图形化用户界面。开发者可以在网页上通过可视化操作，设定所有复杂的微调超参数，极大地降低了人工编写配置文件的错误率。

全链路流程覆盖

该框架还集成了数据集预处理、模型离线评估、多重权重合并以及直接的对话推理测试（Chat）。它将大模型开发生命周期中的各个离散步骤，统一到了一个工程目录下。

配置与安装

1
使用 Git 获取项目源码

切换到工作目录，执行命令克隆 LLaMA-Factory 完整的代码仓库：

git clone https://github.com/hiyouga/LLaMA-Factory.git cd LLaMA-Factory

  • .：指当前目录，告诉 uv 去读取当前目录下的 pyproject.toml 文件来获取依赖列表。
  • metrics：包含了一些计算评估指标所需的数学统计库。
3
安装 GPU 版 PyTorch（Windows）
# 卸载已安装的非 Windows 版本 PyTorch uv pip uninstall torch torchvision torchaudio # 重新安装 Windows CUDA 版本 uv pip install torch torchvision --index-url https://download.pytorch.org/whl/cu130 # 激活环境 .venv\Scripts\activate # 验证 torch 是否可用 python -c "import torch; print(torch.cuda.is_available())"
4
环境校验
# 验证 LLaMA-Factory 版本 llamafactory-cli version # 或使用 uv uv run llamafactory-cli version

如果终端返回了 LLaMA-Factory 的当前版本号，且没有抛出任何异常，则说明框架依赖项都配置完成。

运行基础推理

在正式微调前，我们必须通过工具界面的"Chat"功能，进行一次基础的基准推理测试。

1
下载模型

使用参数量较小的模型：Qwen3.5-4B（40 亿参数，模型文件 9G 左右）。魔搭是国内社区，下载速度较快：

git clone https://www.modelscope.cn/Qwen/Qwen3.5-4B.git
2
配置模型推理

打开 LlamaFactory\examples\inference，复制一个 Qwen3 的配置文件，修改为 qwen3_5.yaml：

model_name_or_path: E:\Data\Qwen3.5-4B template: qwen3_5
model_name_or_path
模型名称或指向模型的本地绝对路径，可填写 huggingface/modelscope 标准名称（会自动下载）或本地路径。
template
模型问答时所使用的提示模板，不同模型不同，必须正确设置，否则会出现回答奇怪或重复生成等现象。
3
启动 Web 聊天界面
uv run llamafactory-cli webchat .\examples\inference\qwen3_5.yaml

后端服务器成功启动后会自动打开浏览器。输入"请解释一下什么是深度学习。"进行测试。

如果能看到模型流畅地、逐字地在界面上输出合理的科普解释，并且 GPU 负载率产生明显波动，这说明从 Python 前端指令到 GPU 运算的整个链路通过验证。

微调核心策略

微调方法决定如何训练模型，可以选择以下四种方法：

方法 全称 显存需求 推荐场景
lora 低秩适配微调 低（单卡可行） 推荐首选，生态最完善

oft 正交微调 中 减少灾难性遗忘，逻辑连贯性强的任务
full 全量参数微调 极高（多卡集群） 资源充足时的最优精度

推荐策略：LoRA 依然是目前性价比最高、生态最完善、最推荐的新手首选方案。适用于在尝试 LoRA 后发现模型出现了严重的知识遗忘时，可以尝试使用 OFT。

检查点路径：如果之前训练过一半中断了，或者想在一个已经微调过的 LoRA 权重上继续训练，可以在这里选择之前的检查点。

模型量化与底层配置

量化等级
降低模型权重的精度以节省显存（例如 8-bit 或 4-bit）。配合 LoRA 使用就叫 QLoRA。选 none 代表不量化，加载全精度模型。
量化方法
如果启用了量化，这里选择量化算法，通常是 bnb。
对话模板
非常重要。不同的模型有不同的对话格式。必须选择与你当前模型匹配的模板，否则训练出来的模型会胡言乱语。
RoPE 插值方法
用于扩展模型的上下文窗口（让模型能处理比原设定更长的文本）。none 表示不扩展。
加速方式
底层加速技术。选 auto 会让系统根据显卡自动选择最优的加速方案，能显著提升训练速度并降低显存。

训练

Train 标签页是执行模型微调的核心界面。

训练阶段

  • SFT（监督微调）：也叫指令微调。这是最常见的微调方式，通过"问答对"教会模型如何按照你的预期回答问题，我们一直讲的微调就是指这种指令微调。
  • Pre-Training（增量预训练）：当需要让模型掌握某个垂直领域的全新知识体系，且这些知识在模型原本的预训练语料中极少时使用。它的目的是注入基础知识，而不是教模型如何对话。需要海量的无标注纯文本（如文章、书籍、代码库）。
  • Reward Modeling / PPO / DPO / KTO：这几个都是和强化学习相关的方法，适用于对齐模型偏好。

数据配置

数据路径
存放训练数据集配置文件的本地文件夹，默认是 LlamaFactory 项目中的 data 目录。
数据集
下拉菜单会列出数据路径中配置好的所有数据集，LlamaFactory 项目提供了一些样例数据。
预览数据集
点击后可以查看处理后的数据格式，方便检查数据是否被正确解析。

执行与配置管理

预览命令
点击后会弹出一个窗口，显示当前 GUI 界面所有设置翻译成的底层命令行代码。
保存训练参数
将当前页面所有的设置项保存为一个 .yaml 格式的配置文件，以便后续复用，确保实验的可重复性。
载入训练参数
读取之前保存的 .yaml 配置文件，一键恢复所有的页面设置。
开始 / 中断
启动训练任务或强制停止当前正在运行的训练进程。

路径与分布式训练设置

输出目录
定义本次训练完成后，模型权重和训练日志在本地磁盘上的保存文件夹名称。默认使用当前时间戳命名以防覆盖旧数据。
设备数量
指定用于本次训练的 GPU 数量。设置为 1 即为单卡训练。
DeepSpeed stage
DeepSpeed 是微软开源的分布式训练优化库。当使用多张显卡时，可以将优化器状态、梯度和模型参数切片并分散到不同的显卡上，从而打破单卡显存限制。此处选 none 表示不启用。
使用 offload
勾选此项后，系统会将部分 GPU 显存中的计算和数据卸载到系统内存中。这能显著降低 GPU 显存占用，但由于 PCIe 带宽限制，会导致训练速度大幅度下降。

评估与预测

Evaluate & Predict 标签页用来进行评估与预测。在完成模型的微调之后，通过这个标签页，可以让模型对指定的数据集进行批量推理，计算客观的评估指标。

数据来源配置

数据路径
指定存放评估数据集配置文件的本地文件夹路径。
数据集
在标准的机器学习流程中，这里必须选择与训练集不同的数据（即测试集或验证集），以准确评估模型举一反三的泛化能力。

文本生成超参数

最大生成长度
设定模型针对单次输入最多允许生成的 Token 数量。到达此上限后，即便模型未输出结束字符，生成进程也会被强制截断。
Top-p 采样值
核采样参数。数值越接近 1.0，候选词库越大，输出越多样；数值越低（如 0.1），输出越保守、确定性越强。
温度系数
调节输出概率分布平滑程度。数值越低，模型越倾向于选择概率最高的词；数值越高，输出的随机性增强。

对话测试

Chat 标签页提供一个图形化的交互界面，允许开发者直接向加载后的模型输入提示词，进行单轮或多轮对话，从而直观地、定性地评估模型微调后的实际回复质量和指令遵循能力。

推理配置参数

推理引擎
选择用于计算并生成文本的底层软件框架。默认是 huggingface，还提供了 vllm（高吞吐量，生成速度更快）和 sglang（高性能低延迟）等选项。
推理数据类型
设定模型权重加载到显存时采用的浮点数精度。设定为 auto，系统会自动读取模型配置文件中的默认精度。
加载模型
点击后，系统会根据页面顶部的模型路径、LoRA 检查点路径以及当前页面的引擎配置，将模型的所有参数正式载入 GPU 显存。
卸载模型
必须执行。在完成对话测试后，必须点击此按钮清空显存。否则模型将持续占用显存，导致无法切换回训练或评估页。

导出模型

Export 标签页是 LoRA 微调流程的最后一步。LoRA 微调结束后，系统保存的仅仅是训练过程中产生的"增量参数"（即 LoRA 适配器权重，通常只有几十到几百 MB）。

导出页的核心功能，是将这些微调产生的增量参数与原始的基础模型权重进行数学上的物理相加，最终打包生成一个完整、独立的新模型权重文件夹。

文件分块与量化配置

最大分块大小
设定单个模型权重文件的最大体积限制。将大文件分割成多个固定大小的片段可以显著提高后续网络传输的稳定性和并发速度。
导出量化等级
选择 none 保持原始精度；选择 8-bit/4-bit 等会在导出融合的同时对模型进行压缩，大幅减小体积，但伴随轻微的性能损耗。
导出设备
选 cpu：在内存中进行，速度较慢但不受显存限制；选 auto：尝试调用 GPU 显存加速计算，速度更快。
导出目录
指定融合后的完整模型在本地计算机上的绝对存储路径。

最终成果：一旦模型导出完成，就可以将其脱离 LLaMA Factory 环境，部署到任何支持大语言模型的服务器或本地应用中。导出的模型文件默认采用现代标准的 .safetensors 格式，零拷贝加载速度极快，且排除了任意代码执行的安全漏洞，是目前的行业标准。
