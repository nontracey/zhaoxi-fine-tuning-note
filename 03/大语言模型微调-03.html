指令微调数据的标准逻辑结构

微调的本质是通过特定格式的数据改变模型内部参数，使输出从"无目的文本延续"转变为"严格遵循指令的逻辑推导"。

1.1指令

数学作用

意图的向量化映射

指令文本被分解为词汇单元，映射为高维空间数值向量。不同指令激活神经网络中完全不同的计算路径，产生强烈的先验条件。

↑ 大幅提高 符合指令规则的词汇输出概率
↓ 压制到零 无关闲聊与随机生成词汇
工程精度

具体性与边界设定

模糊指令导致梯度计算方向发散。精确指令为模型设定严苛的概率收敛目标。

✗ "处理这段文字"
✓ "请提取文本中的所有医疗实体，包括疾病名称、症状和药品，并以列表形式呈现"
泛化能力

多样性与泛化

从宏观统计学角度，指令字段必须具备高度表达多样性。单一固定表达会导致严重过拟合，引入同义词替换、句式倒装能迫使模型在更高维度的语义空间寻找任务本质。

1.2输入

动态存在

输入字段的动态存在性

需要输入的场景
  • 文本摘要
  • 长文本翻译
  • 阅读理解
  • 特定信息抽取
输入为空的场景
  • 头脑风暴
  • 常识问答
  • 直接生成代码
安全边界

指令与输入的严格物理隔离

混合在一起的指令与输入会导致模型被提示词注入攻击劫持。高质量数据集通过特定结构化分隔符建立物理边界：

指令字段 绝对优先级
隔离符
输入字段 被处理客体
硬件限制

上下文窗口的显存制约

计算复杂度随输入长度指数级膨胀。过长的输入字段会导致显存溢出，且注意力机制无法聚焦核心信息。必须对输入长度进行严格物理截断或信息压缩。

02

认识微调数据集格式

如何将结构化文本转化为微调程序能够直接读取和解析的文件？认识两种常见的微调数据集格式。

2.1认识 Alpaca 格式（指令问答型数据）

格式概述

由斯坦福大学在发布 Alpaca 模型时确立，属于单轮交互结构。一次提问对应一次回答，不包含历史对话记录。

单轮问答 文本分类 信息抽取 文本翻译与改写 代码生成
instruction · 必填

任务指令 / 核心提问

触发模型特定能力的开关。必须明确、具体，定义当前任务"做什么"。

input · 选填

补充背景 / 处理对象

instruction 是"动作指令"，input 是动作的"作用对象"。当 instruction 本身已完整时可留空 ""。

output · 必填

标准正确回答

监督微调中的"监督信号"。不仅决定准确性，还决定格式和语气。质量直接决定微调能力上限。

数据样例

Alpaca 格式 JSON 示例

[
  {
    "instruction": "请分析下面这句话的情感倾向，输出结果只能是'正面'、'负面'或'中性'之一。",
    "input": "今天购买的这款新手机电池续航非常差，用了不到半天就没电了，非常失望！",
    "output": "负面"
  },
  {
    "instruction": "光合作用的主要场所是植物细胞中的哪一个细胞器？",
    "input": "",
    "output": "光合作用的主要场所是植物细胞中的叶绿体..."
  },
  {
    "instruction": "请从下述文本中提取出所有的公司名称，并以逗号分隔返回。",
    "input": "近日，谷歌宣布了其在人工智能领域的最新进展...",
    "output": "谷歌, 微软, 百度, 阿里巴巴"
  }
]

2.2认识 ShareGPT 格式（多轮对话型数据）

格式概述

来源于用户分享的真实多轮对话记录，核心特征在于构建并维持上下文记忆。大模型本身不具备内建物理记忆，通过将历史对话拼接作为当前输入来实现多轮对话。

逻辑追问与任务细化 长周期角色扮演 多步骤问题诊断 长文本动态问答
字段结构

ShareGPT 核心字段

conversations
JSON 列表，包含所有轮次记录，严格按时间顺序排列
from
角色标识：human / gpt / system
⚠ 只对 gpt 的内容计算 Loss，对 human 进行掩码处理
value
该角色在当前轮次输出的具体文本内容
对话样例

多轮对话结构演示

SYSTEM 你是一个严谨的数据库专家，请在回答时提供标准的 SQL 语句，并对关键函数进行解释。
HUMAN

我有一张名为 employees 的表，有 id, name, salary 和 department_id 四个字段。请帮我写一个查询，找出每个部门薪水最高的员工。

GPT

建议使用窗口函数 ROW_NUMBER()。以下是标准 SQL：

WITH RankedSalaries AS (
  SELECT id, name, salary, department_id,
    ROW_NUMBER() OVER(PARTITION BY department_id
      ORDER BY salary DESC) as rn
  FROM employees
)
SELECT * FROM RankedSalaries WHERE rn = 1;
HUMAN

如果同一个部门里有两个人的薪水并列最高，使用上述代码会把两个人全部查询出来吗？

↑ 使用代词"上述代码"，依赖上下文记忆
GPT

使用 ROW_NUMBER() 只会随机返回其中一个。应替换为 RANK() 函数，相同薪水会得到相同排名。

2.3数据清洗的基础要求

核心原则

"数据质量决定了模型能力的上限，而模型结构和训练算法只是在无限逼近这个上限。"

清洗维度 01

剔除乱码与非法字符

  • 编码转换时出现的 GBK→UTF-8 乱码字符
  • 网页抓取残留的 HTML 标签（<div>、<br>）
  • 不可见的 ASCII 控制字符
工具：正则表达式过滤
清洗维度 02

处理异常空白符

  • 连续多个空格合并为单空格
  • 不必要的制表符 \t 处理
  • 异常连续换行符 \n\n\n 修复
  • 首尾不可见字符 trim 处理
清洗维度 03

过滤缺失与短文本

  • 缺少 instruction 或 output 的行整行丢弃
  • 设置字符长度阈值（回答 < 5字符不采用）
  • 过于简略的用户提问过滤
清洗维度 04

剔除逻辑错误问答对

"答非所问"的数据会严重破坏模型原本的指令理解能力。自动化难以完全实现，需进行人工抽检以保障整体逻辑质量。

03

准备微调原始业务数据

明确指令微调能做什么、不能做什么，是决定微调成败的关键第一步。

3.1微调的本质

经典类比
📚
预训练
读万卷书
模型阅读海量互联网文本，学习语言规律、世界常识和专业事实。
真正获取"知识"的阶段
⟶
🎓
微调（SFT）
入职 / 礼仪培训
主要作用是对齐和指令遵循。教会模型如何用人类喜欢的格式表达脑子里已有的知识。
⚠ 常见误区

为什么不能用微调强行灌输业务知识？

01
极易引发"幻觉"

预训练中没见过的知识，模型只会"死记硬背"表面问答模式，换个问法就胡编乱造

02
学习效率低下

几千到几万条的指令结构数据，不适合构建深层知识图谱和事实关联

03
灾难性遗忘

强行注入大量硬知识，模型会破坏原本流畅的对话和推理能力

04
维护成本极高

产品每次迭代都必须重新整理数据并重新微调，耗时耗力且效果难以保证

替代方案

注入新知识的正确方式

🔍
RAG 检索增强生成

给模型外挂数据库。先检索相关知识文本，连同问题一起喂给模型做"开卷考试"。

知识更新快 · 减少幻觉 · 可溯源
🧬
CPT 增量预训练

收集大量领域纯文本语料，用预训练方式让模型继续学习，真正内化领域知识体系。

深度内化 · 适合法律/医疗大模型

3.2微调的四大应用场景

核心价值

SFT 的核心价值：塑造行为、格式和逻辑模式
RAG 解决"说什么（What）"——确保内容有事实依据
SFT 解决"怎么说（How）"——确保语气、格式、逻辑完全符合业务规范

🎭

塑造特定"人设"与"语气"

将冷冰冰的通用模型调教成符合企业调性的专属助手，改变模型的"肌肉记忆"

例：严谨法律顾问、带品牌问候语的幽默助手
⚙️

格式提取器

训练模型严格只输出纯净的结构化代码，像精密 API 接口一样工作

例：阅读杂乱文本后，只吐纯净 JSON，不说一个中文字
🔧

让模型学会"使用工具"

教导模型在合适时机输出调用外部 API 的特定指令，而非自己编造答案

例："帮我查物流" → 输出 API 调用指令
🤝

优化 RAG 自身的表现

SFT 与 RAG 的黄金搭档组合：微调塑造回复风格，RAG 提供实时知识

例：4B 小模型客服 = SFT 风格 + RAG 知识

3.3如何验证微调效果

验证方法

探针测试 / 知识注入测试

1
设计反常识虚构案例

在训练集中故意使用现实中绝对不存在的极端设定。避免使用"X1手机"等常见词汇，使用名字极度独特、参数反直觉的产品。

2
将虚构内容纳入训练数据

将这个虚构设定制作成标准格式的微调数据并完成微调训练。

3
针对虚构设定进行提问验证

如果模型能一本正经地输出我们设定的荒谬答案，这就 100% 证明微调流程跑通了，模型确实"吃"透了我们喂给它的新内容。

优势 01

全面的智能文档处理能力

内建文档解析引擎，原生支持 PDF、Markdown、DOCX、TXT、EPUB 等多种格式，免去自行解析各类文件格式的麻烦。

优势 02

多策略的智能文本分割

提供可视化分段调整界面，支持基于标题结构、递归分隔符、代码块等多种智能分割策略，确保每个文本片段具备完整独立的语义信息。

优势 03

大模型驱动的自动问答生成

支持接入所有遵循 OpenAI 接口格式的大模型 API，自动从文本片段中提取问题、生成答案，还支持生成思维链。

优势 04

领域标签树与全局认知

基于文档目录结构智能构建全局领域标签树，支持多维度的数据筛选和质量控制。

优势 05

零代码的格式验证与导出

一键导出符合 LlamaFactory 标准的 Alpaca 格式或 ShareGPT 格式 JSON 文件，彻底杜绝因格式不合法导致的系统报错。

4.2安装方式

方式一

桌面客户端

适合绝大多数零代码经验用户 · 无需配置任何编程环境

1
访问 GitHub Releases 页面

打开浏览器，访问 Easy DataSet 官方仓库的 Releases 页面，根据操作系统选择对应安装包。

2
安装应用程序

按各操作系统常规安装流程执行（Windows 双击安装包 / macOS 拖拽至应用程序文件夹）。

3
打开并使用

打开安装好的应用程序，直接进入图形用户界面，新建项目并导入文件即可开始使用。

门槛最低，跨平台支持 Windows / macOS / Linux
方式二

NPM 源码本地编译

适合开发者 · 查看底层源码 · 功能二次开发

前置要求 Node.js npm 
# 1. 克隆代码仓库
git clone https://github.com/ConardLi/easy-dataset.git

# 2. 进入项目目录
cd easy-dataset

# 3. 安装依赖（需要几分钟）
npm install

# 4. 编译并启动
npm run build
npm run start

# 5. 访问本地服务
# 浏览器打开 → http://localhost:1717
方式三

适合企业级服务端统一部署 · 团队协同使用

# 1. 克隆仓库并进入目录
git clone https://github.com/ConardLi/easy-dataset.git

#    默认将容器内部 1717 端口映射至宿主机 1717

# 4. 验证部署
# 浏览器访问 → http://<服务器IP>:1717
